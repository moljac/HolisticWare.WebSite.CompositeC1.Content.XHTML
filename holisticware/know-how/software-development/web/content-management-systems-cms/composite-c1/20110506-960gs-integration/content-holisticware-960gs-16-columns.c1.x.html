Performance Optimization steps HowTo

Performance 1: Responsivness (Speed) as part of User Experience

Loading pages fast, making pages responsive is crucial part of User Experience UX. Some users can be chased away if it takes too long for the page to load.

Going back to division of product (website) onto:

  • function (functional aspects, purpose) and
  • form ( shape, visual appearance, or configuration of an object)

UX certainly covers both:

  • function:
    • responsiveness
    • usability
  • form: Design

It is not only content that will attract or repel Your visitors, responsiveness influence it too. Even the biggest guys in business like Google and Yahoo suggest that some time should be paid to improving performance of the web site.

Redesigning web site in either in functional sense by adding new features or changing appearance raises the need for performance tweaking. So either optimization is iterative process if design is not stable, or as a recommendation finish functional and formal design and then go for speed optimizations.

This due to the fact that redesigning usually means adding new or changing functionality (let say javascript) and layout (let say css) and those need to be minified (obfuscated), merged (compressed) each time to reduce payload and/or number of http requests.

The greatest guys in business have given us tools for measuring performance of the web site/page. On following urls there is more about rules and tools:

  • Yahoo - yslow http://developer.yahoo.com/performance/rules.html
  • Google - pagespeed at http://code.google.com/speed/page-speed/

Yslow test consist of 3 standard tests with different rulesets from less strict to stricter (Small website or blog, YSlow2 and Classic) and Google's pagespeed has only single test which is quite strict regarding rules.

This makes alltogether 4 tests so our notation for results is quadruple as percents x/100

TotalResult=(ys_small, ys_2, ys_classic, ps)

where:

ys_small - yslow for small site or blog

ys_2 - yslow v 2

ys_classic - yslow with classic ruleset

ps - pagespeed

Results differ because Yahoo insists on some ascpects while Google does not and vice-versa. For example (yahoo takes ETags in account while this is not so important for google)

Performance 2: Test results

Out of the box Composite C1 comes with decent results, for example Starter Kit XSLT based sample site leads to:

TotalResult=(93,80,63,72)

After adding gzip compression (either on web server - iis or for application with HttpModules) and minifcation/merging results are around:

TotalResult=(99,94,90,93)

Adding new features like 960gs layout and SyntaxHighlighter decrease results to:

TotalResult=(89,80,70,80)

This is due to new resources (javascript and css files) needed for implementation. The result can drop even more if files are not hosted on our server, thus increasing redirects.

Some reasons:

  • Composite C1 SyntaxHighlighter fetches js and css from author's (Alex Gorbatschev's) site
  • SyntaxHighlighter generates additional DOM elements thus raising complexity thus forcing more code to be downloaded!.

Workarounds:

  • DOM elements: tradeoff of the beauty (do not use SyntaxHighlighter if page is slow
  • Redirects: host minified/merged resources at Your site

This is excellent proof how form can kill function.
