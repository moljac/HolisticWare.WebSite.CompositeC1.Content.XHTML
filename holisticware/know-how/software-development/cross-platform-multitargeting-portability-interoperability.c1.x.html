Multitargeting - Portability - Interoperability

In human nature there is a drive to automate repetitive tasks in order to be more productive, to spend less time on those tasks, thus gaining more extra time. If this extra gained time is used for further activities/tasks like investment in knowledge (learning, education), cultural hoist/raise, physical activities etc, then one cannot talk that humans are inherently lazy. But on the other hand of this extra spare time is not used for further activities then the labeling "lazy" might be appropriate.
In our team's mindset we encourage team-members to be lazy, to code less through code reuse, to write scripts for long everyday repetitive tasks and to use this time gained for themselves to learn, train, improve themselves in any aspect.
As the matter of fact this kind of "laziness" is considered as added value.

Lazy-bum (admin or/and developer) howto:

  • Script! Write a script!
    After doing something write a script, there's high possibility that it will be needed again (and again).
  • Reuse! Reuse code written!
    When writing code, extract what is useful, parameterize it (generalize) and put into library. Add unit testing to be sure it works after changes (requirements fulfillment, acceptance and regression tests, etc...)

So share code, no matter source or scripts...

History - brief

It is interesting that in Microsoft development community code sharing thoughts intensified with dawn of Silverlight, when it was realized that code between WPF and Silverlight could be shared... Although it was true for other earlier application types (WinForms etc).

With the work of Monoists implementing MonoTouch, Mono For Android etc, code sharing is more than necessity.

Today it is possible to write shared Model (Business Logic) and N Views (Presentation Layers) for N platforms (some can be shared, but generally not)....
