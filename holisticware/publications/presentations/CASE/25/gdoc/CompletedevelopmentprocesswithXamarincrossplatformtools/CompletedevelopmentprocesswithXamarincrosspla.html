Introduction

Shooting star Xamarin on the IT sky tries to establish itself as a leading cross-platform mobile development company, but it is not only mobile though this segment has brought the most attention from developer community. Xamarin tools are based Mono-Project (Mono), open-source technology which implements .net framework and tools for non-Windows (meaning non Microsoft) OS platforms like Linux, Mac OSX, BSD (FreeBSD), Solaris, iOS and Android. So, Xamarin does not only mobile, but desktop and server implementations of .net.

Xamarin 2.0 came out on 2013-02-20 and this version brought a lot of groundbreaking news: completely rewritten and rebranded IDE Xamarin Studio, implementation of c# 5 (.net 4.5) async features on the alpha and beta channel,

Xamarin philosophy and concepts

Xamarin is based on open-sourced project called Mono (Mono project) and keeps maintaining it, but the area in which Xamarin team is earning the money for daily bread is mobile for sure. They claim they have democratized c# as a programming language and brought it to other platforms, while maintaining the ability for novice .net developers to work with Visual Studio (with Visual Studio AddIns) while getting used to new platforms and new tools. In their press releases they say they have brought .net Base Class Libraries (BCL) to other platforms thus enabling .net developers to work with familiar APIs. What was once java, today is c#, maybe even more.

Xamarin philosophy revolves around native programming paradigms, especially in UI design, they have "projected native API - libraries and frameworks into c#" and do the "full native APIs – no compromise", because “write once run many” - does not work with UI - causes unnatural ugliness. So team wraps (binds with bindings) platform’s native APIs (wrapped - similar concepts to p-Invoke, r-p-Invoke). So their recommendation is to work on Windows with WPF, Windows Forms, on MacOSX with Cocoa, on Linux with GTK# on iOS with Cocoa Touch (Xamarin.iOS) and on Android with Android native API (Xamarin.Android).

They are proud  that they have made possilbe to "share code across Android, iOS, Mac, Windows Phone, Windows RT, Windows" and this is not complete truth, because there are Linuxes and other Unixoides. Xamarin emphasises that Xamarin and mono is "Only platform that allows sharing code across these platforms while keeping native performance and UX".

Xamarin 2.0 changes a lot. Xamarin’s impressive SDK brings native iPhone and Android development to C# an .net programmers to Visual Studio.. Ordinary .c# developer can now develop and debug apps for the iPhone, iPod touch, and iPad in Visual Studio in Windows.

On the alpha channel in mono 3.0.x c# 5:

public async Task<int> ExampleMethodAsync()
{
   var httpClient = new HttpClient();

   // The following line activates GetStringAsync, which is an asynchronous method.
   Task<string> contentsTask = httpClient.GetStringAsync("http://msdn.microsoft.com");

   // While the task is active, you can do other work.
   ResultsTextBox.Text += "Doing other work before awaiting contentsTask. . . . .\n";

   // When you await contentsTask, execution in ExampleMethodAsync is suspended,
   // and control returns to the caller, StartButton_Click.
   string contents = await contentsTask;

   // After contentTask completes, you can calculate the length of the string.
   int exampleInt = contents.Length;

   //// You can combine the previous sequence into a single statement.
   //int exampleInt = (await httpClient.GetStringAsync("http://msdn.microsoft.com")).Length;

   ResultsTextBox.Text += "Preparing to finish ExampleMethodAsync.\n";
   // After the following return statement, any method that's awaiting
   // ExampleMethodAsync (in this case, StartButton_Click) can get the integer result.
   return exampleInt;
}

On mobile quality is hard, thus testing is very important.

Shere numbers speak about complexity:

  1. Android:         
  1. 5 OS versions
  2. devices: n * 10^2 (8 vendors)
  3. 27 screen sizes
  4. 20 languages
  1. iOS:
  1. 3 OS versions
  2. 8 devices, but expensive
  3. 20 languages

Mike Shaver Facebook’s Engineering Director: “Top 100 Android devices covered only 1 / 3 Facebook mobile users”. Android fragmentation enormous, number of permutations (permutation matrix) is huge. The most popular Android phone only 9% of the market.

Xamarin pays quite a lot of attention to their users (developers) and through various questionnaires they made some business decisions.

 

Mobile users are picky/spoiled, bug hurt application image thus lowering the ratings and will force users to move to other applications on the store or for enterprise apps engagement goes down and project fails. Another issue is regressions in very short life-cycle of the application. Desktop and web apps usually 6 months, while mobile 6 weeks release lifecycle - mobile is fast moving.

Options for testing

  1. unit tests
    cannot cover all of the code (usually libraries, not UI and user interaction), run in simulators or emulators (workaround with Xamarin cross platform tools)
  2. manual testing
    #1 method used in mobile today
  1. the most simplified version - install app on the device and test it
  2. less simplified - team of testers sending spreadsheets with bugs periodically
  3. advanced - beta testing
    testflight,
    hockeyapp
  4. automated UI testing
  1. BDD behaviour driven development and testing
  2. scripting UI interaction, simulate user interaction, robots doing manual testing

Questionnaires showed the direction for strategic decisions like product types and potential partners in the field of automated UI testing and there was not a lot of options. No one used automated UI testing and only 8% did some testing (mostly) unit testing.

Analysis showed

  1. testlabs expensive and hard to setup (only for Android devices $8k/month), expensive devices, setup, space...
  2. difficult to write sophisticated and realistic scripts
  3. tests are fragile (image recognition)
  4. user interfaces for the tools - very bad

Goal

  1. testing apps on the UI level -  behaviour
  2. test on real devices - hardware  (  device != hardware )
  3. resilient to smal UI changes (widget/control position)
  4. easy (learnin curve), fast jump start, fun to use, UX

Recommended professional testing practices:

  1. unit testing
  2. automated acceptance testing
  3. automated UI (behaviour) testing
  4. continuous builds
  5. continuous testing

Calabash basics

Xamarin Test Cloud is based on Calabash (http://calaba.sh/) automated UI testing script framework based on Cucumber (http://cukes.info/) and Martin Fowler’s Business Radable Domain Specification Language. Similar ideas can be found in NBehave, .net implementation for BDD -  Behaviour Driven Development and Testing.

Clabash is UI Test Automation framework for iOS and Android and it is open sourced. It is written in objective-c, Java and Ruby.and enables testing for native and hybrid mobile apps (can make assertions about web content). Currently it has Ruby API for clients, but c# is currenlty in beta:

[Test()]
public void ShouldBeAbleToPlaceLimitOrders ()
{
   EnterInto ("Username", "cathy@bussiness.com");
   EnterInto ("Password", "xkal38723@2");
   Tap ("Log In");
   AwaitScreenWithTitle ("Place Order");
   AssertButton ("Limit");
   Screenshot (label: "Place limit order");
}

Calabash’s min purpose is tu support BDD (Behaviour Driven Development)        

  1. that tests are written in plain text only
  2. that any person (unskilled) can do it - bad practice to let management write BDD tests

rather: BDD

  1. requires specifications, use-cases, feature-lists
  1. should be readable and understandable by project and team memebers
  1. should improve and encourage communication

Feature lists are written in a formal language (has syntax) called Gherkin (similar to spoken language) and is not programming language (no loops, no conditional statements):

Feature: Trades
 Scenario: Selectable Trading Order Types
   Given I am logged in as "Cathy"
   When I go to the Trading Orders Screen
   Then it should display the Trading Orders order by Id

Architecture:


Client on a host computer (usually script, but can be IDE addin) sends commands and server on the device or emulator performs clicks, events - simulated user interaction. Clients are written in Ruby, Java, objective-c and c#. Client and server communicate with simple JSON protocol - simple http requests and can be written in any language. Server internals work different on Android and iOS and is written in java and objective-c and deeper knowledge is necessary only if extensions of Calabash are intended. Internally

Calabash is scripting language for Xamarin Test Cloud which works in 2 modes;

  1. AppExplorer mode where test and scripts automagically generated and executed
  2. script mode where user sends predefined scripts with app package

Calabash runs localy (normally) but can be easily be integrated in CI systems like TeamCity or Jenkins.
